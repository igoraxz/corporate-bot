## Common Pitfalls

- **FACTS_UPDATE in multi-turn flows**: The model may emit FACTS_UPDATE in an intermediate TextBlock (before a tool call), then produce a different final response. `final_text` only holds the last TextBlock / ResultMessage. Fact extraction uses `all_text_blocks` (accumulated from every turn) to catch all FACTS_UPDATE emissions. Fixed in commit after `4f67d17`.
- **SDK `receive_response()`**: Takes NO arguments. Don't pass `include_partial` or similar
- **SDK `callable` bug**: Python 3.12 doesn't allow `callable` as type hint — use `typing.Callable`
- **Cross-SDK state**: Per-session state dict (`_session_state` keyed by `session_key`, NOT ContextVars — those don't propagate to SDK hook tasks). Four isolation layers: (1) **Hooks**: `build_hooks(session_key)` wrapper closures eagerly register `session_id → session_key`, then `_get_state_for_hook()` resolves via `_sid_to_skey` only. (2) **Send tools**: per-client wrappers via `create_messaging_server(session_key)`. (3) **Memory tools**: per-client wrappers via `create_memory_server(session_key)`. (4) **Service/deploy tools**: per-client wrappers via `create_services_server(session_key)`. Must call `set_reply_threading(session_key=...)` before each query. All public getters/setters require explicit `session_key` — see AP-15 in coding-principles.
- **`_active_session_key` global — REMOVED**: The process-global `_active_session_key` was eliminated. All session identity is now resolved via explicit `session_key` parameters (callers) or `_session_key` in MCP tool args (injected by `_wrap_tools_with_session_key`). See AP-15.
- **AccessContext data isolation**: `allowed_chat_ids` is set to `(chat_id,)` (single chat) for non-admin sessions AND admin in group/topic chats. Only admin in private DMs gets empty `allowed_chat_ids` (global scope). `has_filter` property returns True unless admin DM. This scopes ALL memory/facts/media/RAG queries. AccessContext construction is security-critical.
- **Entrypoint permissions**: New volumes/dirs need `chown botuser:botuser` in entrypoint.sh
- **Docker file mounts**: NEVER add bind mounts for files that may not exist on host — Docker creates them as directories, causing OCI mount errors. Use named volumes for mutable data
- **Playwright**: Runs as a shared SSE server (started in `main.py` lifespan on port 3001). SDK clients connect via `mcp_size_guard.py` SSE→stdio bridges — each bridge gets a separate tab in the shared browser. Profile persisted at `/app/data/browser-profile`. Stale `Singleton*` lock files are cleaned on startup. Config: `PLAYWRIGHT_MCP_PORT` env var (default 3001)
- **MCP Size Guard**: All browser MCP servers (Playwright, Camoufox) are wrapped with `bot/mcp_size_guard.py`. Two modes: `wrap` for stdio servers, `sse` for SSE→stdio bridging. Features: (1) text truncation at `MCP_MAX_RESPONSE_BYTES` (default 500KB) to prevent SDK 1MB buffer crash, (2) **image-to-file**: ALL base64 images are saved to `data/tmp/<prefix>-<ts>.png` (prefix configurable via `MCP_IMAGE_FILE_PREFIX`, default "screenshot") and replaced with `[Screenshot saved to file](path)` — matches Playwright's `--image-responses omit` behavior, (3) chunked pipe reads (64KB) to prevent stdio deadlocks on large responses, (4) per-tool timeout (120s default), (5) endpoint URL validation (same-host check), (6) **request injection**: auto-injects viewport 1920x1080 + light colorScheme into `create_tab` calls (Camoufox defaults to 1280x720 dark)
- **Camoufox**: Anti-detect Firefox browser via `camofox-browser` Docker service. Set `CAMOFOX_API_KEY` in `.env`. Default URL: `http://camofox-browser:9377` (override with `CAMOFOX_BROWSER_URL`). Used for sites that block Playwright (airlines, flight aggregators, anti-bot protected sites). **Dual persistence model**: (1) **Browser-native storage** — cookies/sessions in the `camofox-data` Docker volume (`/home/node/.camofox`) survive container restarts even without explicit profile saves. OAuth sessions (Google SSO, etc.) persist here automatically. (2) **Explicit profiles** — `camofox-mcp` save/load profile mechanism stores cookie snapshots as JSON files. These are a *separate layer on top* of native storage — useful for cross-session portability but not required if the browser container is stable. **Chat-level isolation**: profiles dir is per-session — admin chats use `/app/data/camofox-profiles/team/`, non-admin chats use `/app/data/camofox-profiles/{chat_id}/` (path-traversal sanitized).
- **Microsoft 365 MCP (Softeria)**: Outlook calendar + email via `@softeria/ms-365-mcp-server@0.54.1` npm package (stdio, npx auto-install). Pin version — review periodically (last: 2026-04-02). Delegated permissions with client secret for secure token refresh. `--enabled-tools` regex limits to read-only mail + calendar + user info (scopes: Mail.Read, Calendars.ReadWrite, User.Read). **Do NOT use `--preset`** — it overwrites `--enabled-tools` in cli.js (preset pattern runs after Commander parse). Token cache at `/app/data/ms365-mcp/msal_cache.json` (persistent volume). Config: `MS365_MCP_CLIENT_ID`, `MS365_MCP_TENANT_ID`, `MS365_MCP_CLIENT_SECRET` in `.env`. Conditionally enabled — missing env vars = server not registered. MSAL cache filenames must avoid "token"/"secret" (hooks.py blocklist).
- **M365 first-time login**: MUST be run from HOST MACHINE (not container) — container security hooks block env var access and MCP session lifecycle kills polling. Script: `python3 scripts/m365_login.py` from repo root. It reads `.env` directly, polls Azure AD device code flow, builds MSAL cache in **Softeria envelope format** (`{_cacheEnvelope: true, data: "<MSAL JSON>", savedAt: <ms>}`), `docker cp`s it into the running container, verifies via Graph API call, and restarts the container so MCP server loads the new cache. Commands: `python3 scripts/m365_login.py` (login), `check` (test credentials), `verify` (verify cached token in container). Resumable (saves state to `data/m365_auth_state.json`). **Token refresh is automatic**: Softeria auto-refreshes via MSAL `acquireTokenSilent()` + 401 retry, persists refreshed tokens to disk. Re-auth only needed if refresh token expires (90 days inactive). **Cache format**: `selected_account.json` must use `{accountId: "..."}` (NOT `homeAccountId`), wrapped in Softeria envelope.
- **M365 dual cache path mismatch (known bug)**: `scripts/m365_login.py` writes cache to `/app/data/ms365-mcp/` (`CONTAINER_CACHE_PATH`), but `mcp_config.py` reads from `/app/data/credentials/ms365-mcp/` (`ms365_cache_dir`). After any re-auth or token refresh issue, you must copy files between both paths: `cp /app/data/ms365-mcp/* /app/data/credentials/ms365-mcp/`. TODO: align the paths in a future commit.
- **M365 stale MCP subprocess after re-auth**: Softeria MCP server starts once per SDK session and caches the token at startup. After copying fresh cache files, the running session still uses the OLD token. Fix: `/reset` to restart the SDK subprocess with fresh MCP connections. This is the most common cause of "AUTH_ERROR even though verify-login works on the CLI". Full refresh procedure: (1) copy fresh cache files to both paths, (2) `/reset` the session.
- **Kiwi Flights MCP**: Remote HTTP server at `https://mcp.kiwi.com` (override with `KIWI_MCP_URL`). No API key needed. Official MCP server by Kiwi.com + Alpic AI. Tool: `search-flight` (one-way/round-trip, ~750 airlines, self-transfer routing). Returns curated options with booking links.
- **Google Flights (fli)**: Custom MCP tools `google_flights_search` and `google_flights_search_dates` in mcp_tools.py. Calls fli's search engine directly (the `fli-mcp` stdio server is broken with fastmcp>=3.0). No API key — reverse-engineers Google Flights Protobuf API. Currency default: GBP. Uses `curl-cffi` with browser TLS fingerprint impersonation. Rate limited to 10 req/s with retry/backoff. Concurrency capped at 8 via `_GFLIGHTS_SEM` semaphore (Google 429s at ~9 concurrent).
- **Bubblewrap (bwrap) sandbox setup**: Requires THREE host-level configs in docker-compose.yml: (1) custom seccomp profile `seccomp=./seccomp-bwrap.json` (allows clone/unshare with NEWUSER|NEWPID|NEWNET|NEWNS + mount/umount2/pivot_root), (2) `apparmor=unconfined` (docker-default blocks mount propagation), (3) `no-new-privileges:true` + `cap_drop: ALL` (NO `cap_add: SYS_ADMIN` needed). Common failures: "No permissions to create new namespace" = missing seccomp profile; "Failed to make / slave" = AppArmor blocking mount; "execvp bash: No such file or directory" = missing `--symlink usr/lib64 /lib64` (dynamic linker); "cannot open shared object file" = missing `--ro-bind /etc/ld.so.cache /etc/ld.so.cache`. Always test via `sandbox-exec.sh` (handles merged-usr detection, /etc binds) not raw bwrap. `--proc /proc` deliberately omitted — kernel blocks proc mount in containerized user namespaces; no /proc = more secure (no PID enumeration, no env var leaks). Staging: seccomp profile must be copied to staging repo separately (`cp seccomp-bwrap.json ~/${COMPOSE_PROJECT_NAME}-staging/`).
- **Skills discovery**: SDK needs `cwd="/app"` + `setting_sources=["project"]` to find `.claude/skills/`. Skills symlinked from `/host-repo/.claude/skills/` to `/app/.claude/skills/` in entrypoint.sh
- **Prompt file placeholders**: Use double braces `{{` `}}` in prompt .txt files for literal braces (Python `str.format()` interprets single braces as placeholders)
- **System prompt size limit**: ~128KB (Linux `MAX_ARG_STRLEN`). The SDK passes the system prompt as a single CLI arg to the Claude Code subprocess. Base prompt is ~57K chars. Inline module content for non-team chats is trimmed first if the limit is approached. Exceeding 128KB causes `[Errno 7] Argument list too long` crash.
- **Bot logs**: `/app/logs/bot.log` (RotatingFileHandler, 10MB, 5 backups). DEBUG to file, INFO to stdout. Persists across container restarts. Use Read/Grep to search. Previous container's crash is typically at the end of `bot.log.1`.
- **Pyrogram session files**: `.session` files in `tg_session/` contain MTProto auth keys (full Telegram account access). Protected by hooks.py file blocklist. Never log, commit, or expose them.
- **No inline keyboards**: User accounts (Pyrogram userbot) cannot create inline keyboards (Telegram server-side restriction). The bot presents choices as numbered text options and the LLM interprets replies naturally.
- **Pyrogram + FastAPI coexistence**: Use `await client.start()` / `await client.stop()` in FastAPI lifespan. Do NOT use `client.run()` (conflicts with Uvicorn's event loop).
- **Env var stripping — Bash tool can’t see sensitive config**: `config.py` uses `os.environ.pop()` to strip sensitive vars (`GEMINI_API_KEY`, `GOOGLE_OAUTH_CLIENT_SECRET`, `CAMOFOX_API_KEY`, `TOKEN_ENCRYPTION_KEY`, etc.) from the process environment after reading them into module-level constants. The SDK subprocess (Bash tool) inherits the stripped env — so `os.environ.get("GEMINI_API_KEY")` from Bash returns empty even when the value is set. **To verify config**: check startup log warnings (absence of "not set" warning = key is configured), or use module-level constants (`from config import GEMINI_API_KEY`). NEVER check sensitive config via `os.environ.get()` in Bash — it will always be empty by design. Same applies to `/proc/*/environ` which is blocklisted for the same reason.
- **Workstation channels have NO bot-local harnesses**: Workstation channels use employee Mac-side model/effort/context config from the SDK daemon — NOT from `chat_harnesses.json`. Never show harness tags in `/relays` (workstation channels) menu. The Mac daemon's config (model, effort, context window) is fetched from the relay session metadata. If you need to display config for a workstation channel, show the Mac-side values (model/effort/context) — not bot harness labels.
