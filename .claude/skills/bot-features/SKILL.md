## Key Features

- **Message auto-splitting (TG)**: Messages >4096 chars are auto-split at paragraph/line/sentence boundaries by `telegram.py:_split_message()`. No manual splitting needed.
- **Zero truncation**: All message storage is unlimited — no `[:N]` caps on messages, bot responses, or phone transcripts.
- **Parallel session file awareness**: When multiple admin coding sessions edit the same file in `/host-repo/`, PostToolUse injects a collision warning via `additionalContext`. Each session tracks modified repo files in `modified_repo_files` (set in session state, lazily initialized). `check_repo_conflicts()` in `session_state.py` scans other sessions for overlapping files. Warning-only — never blocks edits. Automatic cleanup via `cleanup_session_state()`. Colocated with deploy gate reset in PostToolUse (shares the `/host-repo/` detection site).
- **Auto media descriptions**: Every incoming media file (images, PDFs, video, audio/voice, text/CSV, office docs) is analyzed by Gemini Flash (`integrations/gemini.py:describe_media()`, orchestrated by `bot/storage/media.py`). Descriptions stored in `media_cache.description`, indexed via FTS5 UPDATE trigger. Startup backfill handles crash recovery. Config: `MEDIA_DESCRIPTION_ENABLED` (default true), `MEDIA_DESCRIPTION_BACKFILL_LIMIT` (default 20), `MEDIA_DESCRIBE_MAX_SIZE_MB` (default 100), `MEDIA_DESCRIBE_INLINE_SIZE_MB` (default 20), `MEDIA_DESCRIBE_TIMEOUT_S` (default 120), `MEDIA_DESCRIBE_MAX_CONCURRENT` (default 3).
- **Synchronous voice transcription**: Voice/audio messages are transcribed by Gemini Flash BEFORE agent processing (`integrations/gemini.py:transcribe_voice()`). Transcript appears in the agent prompt as `Voice transcript: "..."`. Dedicated verbatim prompt (not the search-indexing description prompt). Transcript reused as media_cache description (skips redundant background Gemini call). Config: `VOICE_TRANSCRIBE_TIMEOUT_S` (default 60), `VOICE_TRANSCRIBE_MAX_CHARS` (default 20000). Graceful fallback on failure — agent processes message without transcript.
- **On-demand skills**: 17 Claude Code skills loaded by SDK only when relevant (phone calls, browser, email, travel, PM, coding, etc.). Base prompt is ~57K chars; skills add content on-demand without bloating the system prompt. Skills can also be pre-loaded via harness `claude_md_modules`. Skills awareness prompt ensures bot knows when to invoke each one.
- **Session metadata**: source, chat_id, is_admin embedded in system prompt — survives context compaction so the bot always knows where to reply.
- **Compaction notifications**: When SDK compacts context, bot notifies the user with token metrics (before/after/freed). Three-layer tracking: (1) `get_context_usage()` post-query for authoritative SDK token counts, (2) JSONL transcript parsing for mid-query early notification at first AssistantMessage after compaction, (3) token-drop fallback detecting >40% context decrease between queries. Pre-compaction tokens saved via `save_last_input_tokens()` on every query.
- **Background task notifications** (`agent.py`): SDK emits `TaskStartedMessage`, `TaskProgressMessage`, `TaskNotificationMessage` while `ResultMessage` is held back during background task execution (`run_in_background=true` on Bash/Agent). The message loop handles all three: routes them through `on_tool_status` callback to (a) update `last_sdk_activity` (prevents false zombie detection during long background waits), (b) show `bg:<description>` in the streaming placeholder tool chain, (c) log task lifecycle. Prompt rule in `02_processing_flow.txt` provides behavioral defense-in-depth (model notifies user before going idle).
- **TG fallback salvage**: If the model runs tools but forgets to call send_message, response_text is delivered from the finally block (prevents silent failures).
- **FACTS_UPDATE from all turns**: `all_text_blocks` list accumulates every TextBlock from every SDK turn. Fact extraction (`_persist_memory_updates`) runs on the joined text of ALL blocks, not just `final_text` (which is only the last turn). This prevents FACTS_UPDATE loss when the model emits facts in an intermediate turn then continues with tool calls.
- **Per-chat data access isolation** (`memory.py`, `rag.py`): All data queries (messages, facts, media, RAG chunks) are scoped via `AccessContext` — a frozen dataclass with `allowed_chat_ids` (set of chat IDs the session may access). Only admin in private DMs gets global scope (empty `allowed_chat_ids`); admin in group/topic chats gets chat-scoped data (`allowed_chat_ids=(chat_id,)`) just like employees. Non-admin always gets own-chat only. Enforced at the SQL layer via `WHERE chat_id IN (...)` filters plus `has_filter` property. AccessContext is set per-session in `agent.py` before each query, derived from RBAC execution scope + chat type (private vs group).
- **Scheduled task AccessContext** (`scheduler.py`, `agent.py`): Each scheduled task stores an optional `chat_id` — the originating chat it was created from. At runtime, `process_system_task(chat_id=...)` builds session_key `system:{chat_id}` and determines access: empty/ephemeral → admin (built-in tasks see all facts), admin DM → admin, team group → team-scoped, external → own-chat-only. Source explicitly added from chat_id format (WA JIDs contain `@`). Tasks created via `manage_scheduled_task` MCP tool auto-capture the session's chat_id.
- **Pyrogram connection watchdog**: Scheduler checks TG connection every ~2.5 min via `is_connected()`. Auto-reconnects if dropped, re-registers message handler (Pyrogram clears handlers on `stop()`). Health endpoint includes `tg_connected` field.
- **Failed deploy log capture**: Host watcher captures last 300 lines of container logs + container state to `deploy/last_failed_startup.log` before auto-rollback. `deploy_status` tool includes these logs so the bot can self-diagnose failed deploys without SSH.
- **forward_origin compatibility**: `pyrotgfork 2.x` lacks the `forward_origin` attribute present in newer Pyrogram. Uses `getattr()` with fallback to legacy `forward_from`/`forward_sender_name`/`forward_from_chat` fields.
- **Mid-task message injection**: When a text message arrives while the SDK is actively processing a query for the same chat, it's injected directly via `client.query()` (writes to subprocess stdin via `transport.write()` with `_write_lock`). The SDK picks it up at the next listening checkpoint between tool rounds. On injection: old streaming placeholder is DELETED, new "🧠 Noted..." placeholder created as reply_to the injected message. Media messages skip injection (need full processing). Fallback: if injection fails, message is enqueued normally. **Teardown guard**: injection is SKIPPED when `has_pending_teardown(session_key)` returns True (deferred session reset or budget disconnect pending) — the query is about to finish and the session will be wiped, so injected text would be silently lost. Instead, the message falls through to normal enqueue.
- **Markdown → TG HTML converter** (`bot/md_to_tg_html.py`): new module that converts Mac Claude's native markdown output to Telegram HTML parse-mode. Used by workstation catchup (`_send_desktop_catchup`) and workstation live-streaming (`_handle_relay_message` preview + final text). Pure stdlib, no new deps. Scheme-checked `<a href>` (http/https/tg/mailto only). Tolerant of unclosed fences / unmatched emphasis. Mac Claude now writes plain markdown (renders natively in Desktop.app) — bot converts for TG. Prelude `_RELAY_QUERY_PRELUDE` simplified to drop the old "TG_HTML reply mode" instruction.
- **HTML-aware split** (`integrations/telegram.py:_split_message`): opt-in via `TG_HTML_AWARE_SPLIT=1` env. When enabled, long catchup digests split safely at `<blockquote>` boundaries (closes on left chunk + reopens `<blockquote expandable>` on next) so TG never receives unbalanced tags.
- **Workstation catchup UX (Apr 18 2026)**: `_send_desktop_catchup` renders a compact header + topic TOC + per-pair `<blockquote expandable>` detail. Each turn carries timestamp (relative "2h ago" / absolute) and tool-chain summary (`🔧 bash → read → edit`, max 5 shown). Daemon-side `_parse_catchup_delta` uses REVERSE-SCAN on first-attach — seeks blocks backwards from EOF until exactly 10 user-led pairs are assembled (or file start / 50 MB safety cap hit). Replaces previous 2 MB byte-tail heuristic which could miss pairs on tool-heavy JSONLs. Per-line 1 MB DoS guard before `json.loads`. Back-compat: old daemon without `ts`/`tools` fields falls back to the simpler layout.
- **Employee workstation — persistent SDK daemon**: Workstation-mode chats bypass the bot's SDK entirely — text messages acquire a client from `bot/relay/pool.py`, which keeps one long-lived daemon per workstation channel alive on the employee Mac (spawned via `relay-sdk-persistent`, pidfile at `~/.relay/daemons/<chat_id>.pid`). Pool state machine: DISCONNECTED→CONNECTING→READY⇄BUSY→ERROR→RETRY→DEAD with exp-backoff (1/3/9/27/60s, max 5). Query correlation via uuid4 ids. `create_relay_group` eager-spawns the daemon in the background (fire-and-forget `get_pool().acquire + ensure_ready`) so the first-run catchup digest fires within ~5-15s of group creation without waiting for user input. If the eager-spawn fails (Mac offline, SSH auth), the first user message retries through the normal lazy-spawn path. Subsequent messages reuse the live daemon (no subprocess-spawn cost per turn). Stream-json events flow back through the same long-lived SSH pipe. Streaming placeholder UX mirrors normal chats: `🧠 Thinking... (Xs)` → text deltas accumulate → final reply with `cost $X, tools: N` footer → `⏳ Still working... (Xs)` heartbeat after 60s silence. Mid-task injection: NOW at team-chat parity via stdin-open injection. A fresh user frame arriving mid-flight of a prior query is written to the SAME live `claude --print --input-format stream-json --resume UUID` subprocess's stdin (stdin stays open across queries). Claude processes it as a new turn at its next input checkpoint between tool rounds. Prior query keeps running; its in-flight tools are preserved; both replies stream back to TG with separate placeholders. Daemon-side FIFO correlation (`_pending_qids` deque + per-qid `asyncio.Event`) routes stream/result entries to the correct qid; on `result` the head qid pops and its event fires. Subprocess exits via graceful drain (5 min idle timeout after last result — cancelled by any fresh inject or stdout activity from the CLI). CLI reentrance verified Apr 17 2026 (fact `relay_cli_stdin_reentrant`). **Injection merge semantics verified Apr 18 2026** (fact `relay_injection_semantics_verified`): the CLI's merge-vs-split behavior on a mid-flight stream-json frame depends on the parent turn's mode. **Case A (single-turn text-only)**: frame 2 arrives during a pure text-streaming turn → CLI queues it → emits a SECOND `result` event → bot emits 2 separate replies (one per qid). **Case B (multi-turn tool-using)**: frame 2 arrives between tool rounds → CLI absorbs it into the current AgentLoop → emits ONE `result` with `turns=N+1` → daemon reader pops only qid1 on the single result, and **qid2 stays dangling in `_pending_qids`**, leaving its bot-side placeholder unfinished ("stuck placeholder" bug). Case B is the COMMON case (most team-chat turns use tools). Fix shipped (two-step refinement): initial fix used `num_turns>1` as the merge signal but this was ambiguous — a single-frame parent turn with multiple tool rounds also reports num_turns>1. Refined fix (the one in prod): daemon tracks `_user_text_in_window` — the count of user-role stream-json entries with non-empty TEXT content seen since the last `result` event. On `result`, drain `extras = min(user_text_count - 1, len(pending_qids))` head qids with synthetic `result` subtype=`merged_into_prior_qid`; bot-side deletes those placeholders. Tool_result entries (also `type=user` in CLI output) are excluded because `_extract_text` returns `""` for content lacking a `type=text` block. Window counter resets to 0 at start of each result handler. Case A fix (B2 cancel+restart) is deferred — narrow edge case, higher complexity. `create_relay_group` MCP action creates TG groups linked to Mac Claude sessions by auto-pinning the newest JSONL for the project (optional `pinned_uuid` arg overrides) — falls back to unpinned with a clear note if no JSONLs exist. Bot `--resume`s that UUID so Desktop Claude.app turns interleave naturally. Admin-only (`TG_ADMIN_CHAT_IDS`). Streaming placeholder parity with team chat: `🔧 bash → read → edit` tool-chain states (5s refresh when pre-text, 1 Hz during streaming), `🧠 Thinking…` / `⏳ Still thinking…` fallback states mirror `_build_tg_status`. **Finalize parity (Apr 18 2026)**: placeholder is a STATUS surface only — at every exit path (daemon_error / final_text / empty-text) the placeholder is DELETED and the reply is sent as a FRESH `telegram_send_message` threaded under the user's original via `reply_to_message_id`. Mirrors team-chat lifecycle: placeholder = streaming status, final reply = fresh tool-sent message. TG auto-split handles oversized bodies transparently. Legacy tmux-TUI transport (send_encoded/capture_pane/tail-f/fswatch/interactive buttons/terminal proxy) deleted in Session C (commit ad6a496).
- **Employee workstation — three-scenario catchup parity (Apr 18 2026)**: The daemon's `_emit_desktop_catchup` (called once from `_DaemonSession.__init__`) has two modes selected by saved offset state. **first_run** (no saved offset for this chat+uuid): parses full JSONL, surfaces last 10 query/response pairs, baselines offset at EOF — covers (a) brand-new relay chat init and (b) re-pin to a different UUID (since `_load_offset` is keyed by chat+uuid, a new UUID returns None). **delta** (saved offset exists): parses offset..EOF, surfaces last 10 NEW pairs since saved offset — covers daemon respawn after offline/manual kill. Daemon emits `cause` field in the `catchup` stream-json event ("first_run" or "delta"). Bot refines the cause using `_recently_repinned_chats` / `_recently_recovered_chats` (60s TTL) populated by `mark_relay_repinned()` (called from `/session pin` after successful re-pin) and `mark_relay_recovered()` (called from watchdog post-online kick). Headers: `📱 New relay chat — last 10 turns`, `📌 Re-pinned to new session — last 10 turns from this JSONL`, `🟢 Mac back online — last 10 new turns from Desktop`, `📱 Desktop activity since last sync` (generic delta). **Smart catchup dedup (Apr 22 2026)**: bot tracks `_catchup_last_delivered_ts` per chat (ephemeral dict, latest delivered ISO timestamp). On repin, entries with ts ≤ last delivered are filtered — prevents duplicate delivery when repinning to a JSONL with overlapping history. Empty after dedup → brief "📌 Re-pinned — no new turns" header. `/session pin` auto-kills the old daemon + releases pool client so next message respawns with new UUID. Watchdog `_kick_relay_daemons_for_post_online_catchup()` fires only when Mac was offline ≥60s (avoids cascading respawns on flaps); for each relay chat: kill daemon, release pool, tag recovery → next message respawns with delta catchup. The 60s refinement window is the named constant `_CATCHUP_REFINEMENT_TTL_S` in `main.py` — also drives opportunistic GC of the recent-event dicts. **Missing-JSONL warning (QA case 2 fix)**: when the pinned UUID has no JSONL on Mac (typo, deleted file, etc.), the daemon emits a `catchup_warning` event (subtype=`missing_jsonl`) instead of silently exiting; the bot surfaces a ⚠️ TG message prompting the user to `/session list` and re-pin. Routed through the pool alongside `stream`/`result`/`error`/`catchup`.
- **Employee workstation — always-on realtime Desktop→TG catchup**: A background tailer inside each `sdk_persistent_daemon.py` polls the pinned JSONL every 500 ms and emits `desktop_turn` events for any user/assistant turn written by Desktop Claude.app. The bot-side `RelayPersistentClient` has a per-client listener task that drains the `_desktop_turn_queue` and calls a module-level forwarder registered by `main.py.lifespan` (`_forward_realtime_desktop_turn`). Dedup is three-layer: (1) the daemon collects `message.id` from every stream-json `user`/`assistant` entry it sees during `_spawn_and_stream` into a bounded FIFO `set` (cap=10_000, explicit popleft + discard, NOT `deque(maxlen=)` which would silently break set sync). (2) Fallback offset-range dedup: each query pushes `[start_size, None]` into `_own_write_ranges` at inject-time (in `_ensure_proc_and_write`), and the reader loop finalises it to `[start_size, end_size]` when the matching `result` entry pops that qid from the FIFO. Open ranges from any crashed/unfinished queries are force-closed in the reader loop's post-EOF cleanup. Dedup order is id-first: entries with `message.id` matching own set → skip; entries with id NOT in set → emit regardless of offset; entries without id → consult `_is_own_write_offset`. (3) **Bot-side crash-recovery dedup** (`relay_ux.py`): `BoundedDedup` per-chat FIFO set (cap 10K) checks `message_id` in `_forward_realtime_desktop_turn()` before TG delivery. If daemon crashes between `_emit()` and `_save_tail_offset()`, re-delivered entries are caught by this layer. Ephemeral (in-memory) — persistent dedup unnecessary since daemon+bot compound crash is near-zero probability. Entries without `message_id` pass through (old daemon format, metadata). Cleanup on relay deletion via `clear_desktop_turn_dedup()`. Tailer uses its own offset file `<chat>.<uuid>.tail_offset` (separate from the per-query `<chat>.offset`) to avoid write race. Per-query `_emit_desktop_catchup` invocation is fully retired — the function now fires ONCE per daemon lifetime from `_DaemonSession.__init__` (Apr 18 2026), emitting last 10 query-response pairs from the JSONL as a TG-only digest. Never injected into model context (JSONL is the model's history via `--resume UUID`). Tailer handles rotation (inode change → emit `jsonl_rotated`, reset offset), truncation (size regression → reset), partial-line defer (UTF-8 safe: finds last `\n`, keeps trailing bytes for next tick), and slice caps (10 MB per tick — emit `slice_skipped` meta on overflow). **Suppressed-offset preservation (Apr 25 2026)**: entries suppressed by the `_pending_qids` guard (Desktop activity during an active bot query) do NOT advance the tail offset — the cursor stays at the first suppressed entry so it survives for catchup on daemon respawn. Without this, bot-side query timeouts caused permanent data loss: the offset advanced past entries that were never forwarded. Slice-cap safety valve overrides suppression cap to prevent unbounded re-reads. No new deps; constants at top of daemon file (`_REALTIME_TAIL_INTERVAL_S`, `_REALTIME_DEDUP_CAP`, etc.).
